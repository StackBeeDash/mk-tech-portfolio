# レート制限設計

## 概要

API の可用性を保護するためのレート制限設計。トークンバケットアルゴリズムとスライディングウィンドウの 2 つのアプローチを使い分ける。

## Why レート制限が必要か

- **DDoS / ブルートフォース攻撃の緩和**: 短時間に大量のリクエストを送る攻撃を防止
- **リソースの公平な配分**: 特定のユーザーが API リソースを独占することを防止
- **コスト管理**: Cloud Run の従量課金において、異常なトラフィックによるコスト暴走を防止

## アルゴリズムの選択

### トークンバケット（Token Bucket）

一般的な API エンドポイントに適用。バースト的なアクセスをある程度許容しつつ、平均レートを制限する。

```
[バケット容量: 100 トークン]
[補充レート: 10 トークン/秒]

リクエスト 1: トークン 99/100 → 許可
リクエスト 2: トークン 98/100 → 許可
...
リクエスト 100: トークン 0/100 → 許可
リクエスト 101: トークン 0/100 → 429 Too Many Requests
（1秒後: トークン 10/100 に補充）
```

**メリット**: バースト対応可能、実装がシンプル
**用途**: 一般的な CRUD API

### スライディングウィンドウ（Sliding Window）

認証エンドポイント等、より厳密な制限が必要な箇所に適用。

```
[ウィンドウ: 1分間]
[上限: 5 リクエスト]

12:00:00 - リクエスト 1 → 許可 (1/5)
12:00:15 - リクエスト 2 → 許可 (2/5)
12:00:30 - リクエスト 3 → 許可 (3/5)
12:00:45 - リクエスト 4 → 許可 (4/5)
12:00:50 - リクエスト 5 → 許可 (5/5)
12:00:55 - リクエスト 6 → 429 Too Many Requests
12:01:01 - リクエスト 7 → 許可 (リクエスト1がウィンドウから外れた)
```

**メリット**: 境界問題がない（固定ウィンドウと異なり、窓の端でバーストしない）
**用途**: 認証 / パスワードリセット / OTP 検証

## エンドポイント別のレート制限設定

| エンドポイント | アルゴリズム | 制限値 | キー |
|--------------|-----------|--------|------|
| `POST /auth/signin` | スライディングウィンドウ | 5 回/分 | IP |
| `POST /auth/signup` | スライディングウィンドウ | 3 回/時 | IP |
| `POST /auth/reset-password` | スライディングウィンドウ | 3 回/時 | IP |
| `GET /api/*` | トークンバケット | 100 回/秒, バースト 200 | ユーザー ID |
| `POST /api/*` | トークンバケット | 30 回/秒, バースト 60 | ユーザー ID |
| `DELETE /api/*` | トークンバケット | 10 回/秒, バースト 20 | ユーザー ID |

## レスポンスヘッダー

```http
HTTP/1.1 200 OK
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 95
X-RateLimit-Reset: 1700000060

HTTP/1.1 429 Too Many Requests
Retry-After: 30
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1700000060
```

## ストレージ戦略

### インメモリ（単一インスタンス）

開発環境やトラフィックが少ない場合:

```typescript
// Map<key, { tokens, lastRefill }>
const store = new Map<string, TokenBucket>();
```

### Redis（マルチインスタンス）

本番環境で Cloud Run が複数インスタンスにスケールアウトする場合:

```typescript
// Redis の MULTI/EXEC でアトミックに操作
await redis.multi()
  .get(`rate:${key}`)
  .pttl(`rate:${key}`)
  .exec();
```
