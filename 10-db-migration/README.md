# 10: DB Migration - スキーマ設計 & マイグレーション戦略

マルチテナント SaaS を題材に、PostgreSQL のスキーマ設計、RLS（行レベルセキュリティ）、pgvector によるベクトル検索、および段階的マイグレーション戦略を実践するサンプルです。

## Why この設計にしたか

### 1. マルチテナントで「共有スキーマ + RLS」を選択した理由

マルチテナントの実現方式には主に3つあります:

| 方式 | メリット | デメリット |
|------|----------|------------|
| DB 分離 | 完全な分離、バックアップが容易 | 運用コストが高い、テナント数に比例 |
| スキーマ分離 | 分離と共有のバランス | マイグレーションがテナント数分必要 |
| **共有スキーマ + RLS** | 運用がシンプル、スケーラブル | RLS の設計を間違えるとデータ漏洩 |

今回は「共有スキーマ + RLS」を採用しました。理由:
- テナント数が多い SaaS では DB/スキーマ分離の運用コストが現実的でない
- PostgreSQL の RLS は DB エンジンレベルで強制されるため、アプリケーションコードのバグによるデータ漏洩リスクを低減できる
- Supabase 等のマネージドサービスとの親和性が高い

### 2. マイグレーション戦略: 段階的適用

マイグレーションを以下の3段階に分けています:

1. **001_initial.sql** - テーブル作成（構造の定義）
2. **002_add_rls.sql** - RLS ポリシー適用（セキュリティの確保）
3. **003_add_vector.sql** - pgvector 拡張（AI/ML 機能の追加）

この順序にした理由:
- 基本構造を先に確立し、その上にセキュリティ層を載せる
- pgvector は追加機能であり、コア機能に影響を与えずに後から追加できる
- ロールバック時に機能を段階的に戻せる

### 3. pgvector を採用した理由

- PostgreSQL 内でベクトル検索を完結させ、外部の専用ベクトル DB を不要にする
- 既存の SQL クエリとベクトル検索を JOIN で組み合わせられる
- Supabase がネイティブサポートしており、インフラ構築の手間が少ない
