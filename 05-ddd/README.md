# 05: Domain-Driven Design (DDD)

マーケティング計測プラットフォームを題材に、DDD の戦略設計と戦術設計を実践したサンプルです。

## Why DDD を採用したか

### 1. ドメインの複雑さが一定以上ある

マーケティング計測プラットフォームでは、キャンペーン管理・トラッキング・レポーティングなど、それぞれ異なるビジネスルールとライフサイクルを持つ概念が存在します。これらを1つのモデルで扱おうとすると、モデルが肥大化し変更が困難になります。

### 2. チーム間のコミュニケーションコストを下げたい

境界づけられたコンテキスト（Bounded Context）を明確にすることで、各チームが責任範囲を持ち、独立してモデルを進化させられます。ユビキタス言語を定義することで、ビジネス側とエンジニアの認識齟齬を減らします。

### 3. DDD を「適用しない」判断基準も重要

- CRUD 中心のシンプルなアプリには DDD はオーバーエンジニアリング
- ドメインエキスパートが不在の場合、ユビキタス言語の構築が難しい
- チームが小さく（1-2人）、コンテキスト分割のメリットが薄い場合

**判断基準**: ビジネスルールが3つ以上の概念にまたがり、かつチームが3人以上の場合に DDD の戦略設計を検討する。

## 戦略設計 → 戦術設計の流れ

1. **コンテキストマップ** でシステム全体の境界を俯瞰する
2. **ユビキタス言語** でドメインエキスパートと共通語彙を定義する
3. **境界コンテキスト** ごとに責務と連携パターンを決定する
4. 各コンテキスト内で **エンティティ・値オブジェクト・集約** を設計する
5. **ユースケース層** でアプリケーションのワークフローを記述する
